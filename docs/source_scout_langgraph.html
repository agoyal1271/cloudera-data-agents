How Source Scout uses LangGraph

02_backend/agents/source_scout/react_agent.py

Not a flowchart — a trace. We follow one real query, “find payment data,” through the actual code and watch the graph state object mutate at each step, while a separate SSE stream feeds the browser. The whole point of LangGraph here is that these two things are decoupled.

First — the words, for a Python dev new to LangGraph

You know Python; you just haven’t met these terms. None of them are new ideas — each is something you already do by hand, given a name. Two libraries are in play, and it helps to keep them apart:

LangChain = talking to the LLM (the ChatOpenAI client and the SystemMessage / HumanMessage objects).   LangGraph = orchestrating the multi-step flow around it (everything in this glossary). Source Scout calls LangChain inside a LangGraph node.
GraphStateGraph

The whole flowchart, declared as data instead of nested function calls. You add nodes, wire them with edges, and LangGraph decides what runs next.

In plain Python: instead of hard-coding a(); b(); c(), you describe “a → b → c” as a structure you can branch and loop over.

Statea TypedDict

One dictionary handed to every node — the shared memory for a single run.

In plain Python: literally a dict with known keys (goal, iteration, messages…). Nodes read it and return changes to it.

Nodean async function

One step of work. It receives the state, does something, and returns a partial dict of just what changed.

async def reason_node(state, writer) -> dict:
    ...
    return {"iteration": state["iteration"] + 1}

In plain Python: just a function. The only convention is the signature (state, writer) and “return what changed.”

Reducera per-field merge rule

How a node’s return value gets merged into state. You choose the rule per key.

# default rule → OVERWRITE
iteration: int                           # returning 2 replaces 1
# add_messages rule → APPEND
messages: Annotated[list, add_messages]  # returning [m] grows the list

In plain Python: dict.update() (overwrite) vs list.extend() (append) — the reducer just says which one LangGraph does for that key.

Edgea wire between nodes

“After node A, go to node B.” A plain edge always leads to the same next node.

graph.add_edge("tools", "reason")  # tools always loops back to reason

In plain Python: the hard-coded “what to call next,” lifted out of the function bodies and made explicit so it can loop or branch.

Conditional edgea routing function

An edge whose destination is decided at runtime by a small function that reads state and returns the name of the next node.

def route(state) -> str:
    if state["action"] == "finish": return "finish"
    return "tools"

In plain Python: an if / elif that returns where to go next.

Superstepone tick of the engine

One unit of progress: run the current node, merge its return into state, then follow an edge to the next node. The trace further down is numbered by superstep.

In plain Python: one iteration of the run-loop that LangGraph executes for you.

writer()StreamWriter

A function passed to each node to push an event out to the caller — separate from state. With astream(stream_mode="custom"), whatever you pass to writer() is exactly what the run yields.

In plain Python: like calling a callback(event) or yield-ing progress, without touching your data.

That’s the whole vocabulary. Everything below just shows these pieces doing their job on one real query.

1 · The one idea to hold onto: two channels

A node is just async def node(state, writer) -> dict. It can talk on two channels at once, and they go to two different places:

The return value  →  the graph

What the node returns is a partial state update. LangGraph merges it into ScoutState and uses it to decide where to go next.

  • Drives routing (route() reads state)
  • Persists across supersteps
  • The user never sees it

The writer() call  →  the browser

Calling writer(...) pushes an event out of astream(stream_mode="custom") — straight to the SSE endpoint and the UI.

  • Pure presentation (thoughts, actions, results)
  • Does not touch graph state
  • Same event shapes the hand-rolled agent emitted

So in tools_node you’ll see writer(_emit("observation", …)) (tell the user) and return {"messages": [...]} (feed the next reasoning step) — same function, two destinations.

4 · Reducer spotlight: why messages is special

add_messages = “append, don’t overwrite”

Every other field in ScoutState follows the default rule: a node’s return value replaces it (that’s why iteration: 2 just overwrites 1). But messages is declared Annotated[list, add_messages], so each tools_node return grows the history instead of clobbering the system prompt + goal. That single annotation is the entire reason the ReAct loop “remembers” prior observations — no manual list-appending anywhere in the code.

5 · How we keep it fast at scale

An agent that loads the whole catalog and pipes it into an LLM works fine at 40 tables and dies at 4,000. Every optimization below has one job: make the work independent of catalog size.

5.1 · Retrieve, don’t enumerate — the core move

✗ Enumerate the catalog

catalog = load_all_tables()      # N rows
hits = llm_filter(goal, catalog) # N rows → LLM

Memory and tokens grow with the catalog. At 100k tables the prompt alone is impossible — this is the “million candidates to the model” failure mode.

✓ Retrieve top-N from the index

hits = catalog_store.search(goal, top_k=12)
# vector index returns the 12 nearest — flat cost

_search_assets(message, top_k=12) in scout_chat · catalog_store.search(goal, types, 50) in the ReAct agent. Work is O(1) in the catalog.

The vector index does the heavy lifting before any LLM runs. The model’s job shrinks from “search 100k tables” to “rank these 12” — and that constant-size job is what bounds latency and cost.

5.2 · Then bound what the model sees (RAG)

The retrieved N are the only thing the LLM ranks, so prompt size is constant no matter how big the catalog grows. The full-catalog path still exists, but only as a fallback when the index is empty — and even that is capped. (Exactly the “million candidates” bug we closed by making retrieval work.)

5.3 · Skip search entirely when you already know the name

async def _resolve_asset(name):
    if "." in name:                      # db.table → known
        return describe_iceberg_table(name)  # ZERO vector search
    # …only fuzzy names fall back to a top-1 index match

A fully-qualified db.table resolves with one catalog describe — no embedding, no vector query. Retrieval is reserved for genuinely ambiguous asks.

5.4 · Retrieval only helps if the index is cheap and correct

These are the fixes from the Qdrant debugging — each is a precondition for the speed above:

a query_points()

qdrant-client 1.18 dropped .search(); the call threw and silently returned 0, forcing the slow full-catalog fallback on every query.

retrieval actually fires

b deterministic uuid5 IDs

IDs used hash(), which randomizes per process → every restart re-indexed the same assets as duplicates.

309 → 45 points · no bloat

c over-fetch ×4 + dedup

Fetch limit·4, collapse duplicate assets, keep best score — diverse top-k even on a dirty index.

distinct, relevant hits

d 768-dim match

Collection was built at 384 while embeddings are 768 → every search rejected. Pinned to the model’s dim.

searches return results

5.5 · Hide the latency you can’t remove

a stream_mode="custom"

Each writer() reaches the UI immediately — the user reads the first thought while the tools still run. Perceived latency ≈ time-to-first-token, not total runtime.

instant feedback

b LLM keep-warm

_llm_keepwarm_loop pings every 240 s, under Ollama’s 300 s idle-unload — the first real query never pays a cold-model stall.

no cold start

c deterministic routing

A clear verb plus an asset already in context skips the LLM router call entirely (det_intent).

one fewer LLM hop

d warm caches

Schema Registry is warm-indexed at boot; repeated discovery goals are served from the intent cache.

skip recompute

5.6 · Cap the compute so a confused model can’t run away

recursion_limit = 2 · MAX_ITERATIONS + 5

Each ReAct turn is two supersteps (reason + tools), so the limit lets the loop run at most MAX_ITERATIONS (15) turns before LangGraph halts it — a hard ceiling on tool calls and LLM hops per request, declared once instead of hand-watched with a counter.

7 · What LangGraph replaced

✗ Before — hand-rolled loop

while iteration < MAX:
    iteration += 1
    t,a,inp = await reason(history)
    if a == "finish": break
    obs = run_tool(a, inp)
    history.append(ai_msg)      # manual
    history.append(human_obs)   # manual
emit_final(...)

You own the loop, the counter, the history list, and the stop logic. Streaming is tangled into the loop body.

✓ After — declared graph

graph.add_node("reason", reason_node)
graph.add_node("tools", tools_node)
graph.add_node("finish", finish_node)
graph.add_conditional_edges("reason", route, …)
graph.add_edge("tools", "reason")  # the loop
graph.add_edge("finish", END)

The loop, the cap (recursion_limit), and history growth (reducer) are declarative. Nodes are small + testable; writer() keeps streaming separate from logic.
